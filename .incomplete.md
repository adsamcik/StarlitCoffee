**CONCRETE STRUCTURAL FINDING — direct memory-map comparison (2026-07-08):**
per the user's question "so what's actually crashing?", captured and
compared `/proc/<pid>/maps` and `dumpsys meminfo` for the `:ml` process
under BOTH conditions at matched points in execution: real Mindlayer (7th
crash reproduction, same tombstone signature as all prior runs) vs. the
synthetic single-process harness (which has never crashed across 33+ runs).

*Ruled out — process memory footprint/size:* the synthetic harness's `:ml`
process was actually LARGER than real Mindlayer's at a comparable/earlier
point — 3.61GB total PSS at 40s (synthetic) vs. 2.82GB at 40s / 3.32GB at
80s pre-crash (real). The synthetic harness's "Other mmap" PSS (1.57GB at
40s) already matched what real Mindlayer only reached right before crashing
(1.57GB at 80s). A bigger/more-complex process is NOT the answer — this
refutes the "process footprint size" hypothesis explored earlier in this
session.

*Ruled out — loaded library set:* both processes mmap essentially the
identical set of ~300+ system/framework `.so` libraries (standard Android
app process libraries); real Mindlayer has a handful of extra GPU/EGL
emulation libs not relevant to the crash path.

***FOUND — a genuine, reproducible structural difference: XNNPACK weight-cache
mmap fragmentation.*** Real Mindlayer maps the model's on-disk XNNPACK
weight-cache files (`gemma-4-E2B-it.litertlm.xnnpack_cache_*` and
`...vision_encoder.xnnpack_cache_*`) as **115 and 48 separate, discontiguous
`mmap` segments respectively** — virtual addresses increase monotonically
across the segments, but the underlying FILE OFFSETS being mapped jump
around non-monotonically (mostly decreasing with occasional jumps back up,
e.g. `0x1b3f9000 → 0x1b276000 → 0x1a27b000 → 0x190fe000 → 0x17f80000 →
0x1b0f2000 → ...`) — i.e., the SAME underlying file is mapped in many
small (~1.5-2MB) chunks at scattered offsets, not one linear read.

The synthetic harness, running the EXACT SAME native `liblitertlm_jni.so`
binary against a structurally-identical cache file, maps each conceptual
cache file as **ONE single contiguous region** covering the whole file
(offset 0 to end) — zero fragmentation, every time, across all 33+ runs.

This fragmentation is **static from process start** — identical segment
counts (115/48) at both the 40s and 80s (pre-crash) captures in the same
real-Mindlayer run, so it is set up at engine-init/first-model-load time,
not something that grows progressively across turns. (The SEPARATE
"Other mmap" PSS growth during the conversation — 1.06GB→1.57GB in real
Mindlayer across that same window — is a distinct runtime phenomenon, likely
genuine KV-cache/context-buffer growth, and is NOT what's fragmented.)

**Why this matters:** the crash is `SEGV_ACCERR` (a permission fault on a
mapped page, read access) inside `memmove`, deep in `liblitertlm_jni.so`.
A file mapped as ~115 small, discontiguous `mmap` regions is a FAR more
plausible source of "a later operation reads across what it assumes is one
contiguous buffer, but actually spans a boundary between two independently-
managed VMAs whose permissions/backing can be changed independently" than a
single clean whole-file mapping ever could be. This is the single most
concrete, mechanistically-plausible lead produced by this entire
investigation — a genuine measured difference between the crashing and
non-crashing configurations, not a ruled-out negative.

**What is NOT yet known:** WHY real Mindlayer causes this fragmented mmap
pattern while the synthetic harness (same native binary, conceptually
equivalent Engine/EngineConfig/Conversation call sequence, same model file)
does not. Candidate explanations, none yet confirmed: (a) something in real
Mindlayer's actual Engine/EngineConfig construction sequence, its
`cacheDir` layout (`.../cache/litert_cache/` subdirectory vs. the
synthetic harness's bare `.../cache/`), or session/lease machinery causes
XNNPACK's cache-loader to do on-demand/per-tensor mapping instead of a bulk
read; (b) a stale/partially-invalidated cache file from repeated real-app
reinstalls this session triggering incremental re-validation reads (though
the vendored `liblitertlm_jni.so` BuildId was byte-identical across all
those reinstalls, making a library-version mismatch unlikely); (c) some
other real-Mindlayer-specific runtime state (thread pool sizing, memory
pressure at model-load time from the OTHER engines real Mindlayer
initializes — OCR, embeddings — that the synthetic harness never loads)
influencing the OS's/runtime's decision to fragment the mapping.

## Session 5 continued — VALIDATED root-cause mechanism, then FALSIFIED it as sufficient (2026-07-08)

Per the user's request to "test and validate we actually found the root cause
and make sure we have minimal repro," ran a full empirical validation of the
XNNPACK cache-fragmentation hypothesis above. Findings, in order:

**1. Checked on-device cache state before touching anything.** The real
Mindlayer model file's mtime (`2026-05-24 15:14:18`) already EXACTLY matched
the epoch embedded in the existing `litert_cache` filenames
(`..._1779628458_...`). The original "accidental mtime mismatch from a
re-copy" hypothesis predicts this SHOULDN'T match if the app were stuck in
Build mode — yet it did, while the app was still fragmenting every run. This
was the first sign the original mechanism (accidental staleness) was wrong.

**2. Ran the crash test 3 more times** (4th real-stack reproduction this
session, 91-127s, byte-identical tombstone/BuildId `cb4057ac4deefa809e9c38a14f65f2bb`
each time) — once reusing the existing cache, once against a manually-wiped
"genesis" cache dir, once again immediately reusing THAT freshly-built,
mtime-matching cache. **All 3 runs showed identical fragmentation (114-117
segments for the main cache, 48-50 for vision_encoder) regardless of whether
a valid matching cache file already existed.** This falsified "Load mode is
reachable given a matching identifier" for this app.

**3. Found the actual mechanism via logcat + source.** Every real-Mindlayer
run logs `"XNNPack weight cache build for '...' started"` and
`"...written to..."` — **never** `"XNNPack weight cache loaded from..."` —
even immediately after `"Deleted 0 stale cache files."` (confirming the
identifier DOES match). The synthetic harness, by contrast, DOES log
`"...loaded from..."` on repeat runs. Root cause of the fragmentation:
**`EngineManager.kt:296-317`** calls
`context.cacheDir.resolve("litert_cache").deleteRecursively()`
**unconditionally before every single cold engine init** — a DELIBERATE,
documented design choice (comment cites Phase-0 spike commit `ad1e199`: a
partially-written cache from a crashed init previously bricked the next
attempt with a misleading "model corrupted" error; the fix trades one
guaranteed re-compile per cold start for that safety). **Mindlayer is

## Why accuracy looks "only ~51%" (read before optimizing)

Two structural facts, both verified in code:

1. **The metric is pessimistic — its ceiling is ~83%, not 100%.**
   `exactAcc = exact / total`, and `total` includes correctly-ABSTAINED cells
   (fields not on the bag that the model correctly left blank). ~17% of all cells
   are correctly-absent, so a *flawless* extractor maxes out near 83%. The honest
   "got present fields exactly right" number is **recall = 63%**. See
   `app/src/sharedTest/kotlin/com/adsamcik/starlitcoffee/test/corpus/QualityReport.kt`
   (metric defs) and `BagFieldScorer.kt` (outcomes).
   - Examples: altitude exactAcc 57% but recall 100%; variety 53% vs 83%; farm 42% vs 70%.

2. **OCR is the real ceiling on messy bags — and the benchmark can't see it.**
   `LlmCorpusBenchmarkTest` scores the **text pass only**, on cached OCR
   fixtures. Same prompt across capture tiers: Q0 (studio) 71% → Q1 51% → Q2 46%
   → Q3 (night/glare) 42% — a 29pp swing driven purely by input quality. The
   model can't extract tokens OCR dropped. The fix is the **vision pass**, which
   the benchmark does not currently exercise.

Genuinely weak fields split two ways:
- Free-text, hard to exact-match, low-stakes (user edits anyway): `name`,
  `roaster`, `tastingNotes`.
- Real model confusion, worth fixing: `roastLevel` (~10/28 wrong), `process`
  (bean-form/decaf leak — e.g. `BOHNEN·EN GRAINS` mis-mapped). Both are
  vision-addressable (marks, layout).

Also: hallucination rose 6%→20% (translate-first abstains less); single runs at
temp 0.1 wobble ±5pp.

## How to run the benchmark (do NOT use connectedAndroidTest)

`connectedAndroidTest` uninstalls the app between runs and wipes the OCR
fixtures. Use the dedicated task:

```powershell
.\gradlew.bat scanBenchmark                # capture OCR fixtures, then benchmark
.\gradlew.bat scanBenchmark -PskipCapture  # reuse fixtures (prompt-only iteration)
```

Report lands in `app\build\reports\scan-benchmark\llm-fixture-quality-report.{txt,json}`.
Full details: `.github/instructions/benchmark.instructions.md`.

Notes:
- Debug app id: `com.adsamcik.starlitcoffee.debug` (test `…debug.test`).
- On-device inference is slow (~30 min/run, ~1–2 LLM calls/bag).
- The `scanBenchmark` task auto-enables the Mindlayer debug auto-accept receiver
  so inference runs unattended.
- Use the `android-emulator-mcp` server for any on-device interaction (claim the
  device first).
